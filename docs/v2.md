# rcode v2 Architecture Proposal

## Goal

Make rcode minimal by changing the architecture, not just by hiding the current
configuration behind a smaller YAML schema.

The v1 design makes the remote CLI responsible for too many things:

- finding the host-side server over LAN or Tailscale
- knowing the SSH host name that the editor should use
- handling fallback hosts and retry behavior
- carrying editor command templates
- supporting legacy, split, and unified config formats

v2 should move toward a bob-style split:

```text
remote rcode CLI
  -> sends an open request to an endpoint
  -> local rcode-server daemon
  -> launches the editor with local knowledge of the remote session
```

The remote CLI should only know where to send the request. The local daemon
should know how to open the editor.

## Non-goals

- Do not grow the unified config format further.
- Do not make LAN/Tailscale fallback the default mental model.
- Do not expose editor command templates in the normal config path.
- Do not auto-mutate config files during normal `rcode` execution.
- Do not copy bob wholesale; borrow the boundary between CLI, daemon, protocol,
  and open-flow.

## Target mental model

```text
remote machine
  rcode .
    POST /open
      endpoint: http://127.0.0.1:17339

local machine
  rcode-server
    receives request
    resolves session/SSH authority
    launches Cursor, VS Code, Zed, or another editor
```

The preferred transport is a forwarded local endpoint, for example:

```text
remote 127.0.0.1:17339 -> local 127.0.0.1:3339
```

This removes the need for the remote side to know the host LAN IP, Tailscale IP,
or fallback host list.

## Minimal config

### Remote CLI config

`~/.config/rcode/config.yaml`:

```yaml
endpoint: http://127.0.0.1:17339
editor: cursor
```

Optional:

```yaml
endpoint: http://127.0.0.1:17339
editor: cursor
token: change-me
session: devbox
timeout: 2s
```

Meaning:

- `endpoint`: HTTP endpoint reachable from the remote machine.
- `editor`: requested editor name. Defaults to the daemon default if omitted.
- `token`: optional bearer token for the daemon.
- `session`: optional session name when multiple remotes are registered.
- `timeout`: optional request timeout. Keep this as a single value, not a retry
  matrix.

### Local daemon config

`~/.config/rcode/server.yaml`:

```yaml
bind: 127.0.0.1:3339
default_editor: cursor
```

Optional:

```yaml
bind: 127.0.0.1:3339
default_editor: cursor
token: change-me
sessions:
  devbox:
    ssh: foxy@devbox
```

Advanced custom editor support should be opt-in:

```yaml
editors:
  custom:
    command: custom-editor ssh://{user}@{host}/{path}
```

Built-in editors should not require templates in user config.

## Proposed package boundaries

```text
cmd/rcode              thin CLI entrypoint
cmd/rcode-server       thin daemon entrypoint

internal/app/rcodecli  CLI orchestration
internal/app/rcoded    daemon orchestration
internal/config        minimal v2 config structs and loaders
internal/protocol      request/response types
internal/client        HTTP client for endpoint calls
internal/server        HTTP routing only
internal/openflow      open remote path use-case
internal/editor        built-in editor adapters
internal/session       session and SSH authority resolution
internal/tunnel        optional tunnel helpers
```

Important boundary: the HTTP server should route requests, but the open decision
should live in `internal/openflow`. Editor-specific behavior should live in
`internal/editor`.

## Protocol sketch

Request:

```json
{
  "path": "/home/foxy/project",
  "editor": "cursor",
  "session": "devbox"
}
```

Response:

```json
{
  "ok": true,
  "message": "opened in cursor"
}
```

Authentication, if enabled, should be a simple bearer token:

```http
Authorization: Bearer <token>
```

## Editor behavior

Built-in adapters should cover common editors:

- `cursor`
- `code`
- `zed`
- `nvim`

The daemon receives a request containing a remote path and session. It resolves
that session to an SSH authority, for example `foxy@devbox`, then asks the
selected editor adapter to open it.

Conceptually:

```text
request:  path=/home/foxy/project, editor=cursor, session=devbox
session:  devbox -> foxy@devbox
adapter:  cursor opens ssh-remote+foxy@devbox /home/foxy/project
```

The remote CLI should not render editor command templates.

## Migration plan

### Phase 1: Add v2 config alongside v1

- Add minimal `endpoint`, `editor`, `token`, `session`, and `timeout` config for
  the CLI.
- Add minimal `bind`, `default_editor`, `token`, and `sessions` config for the
  daemon.
- Keep v1 config readable for compatibility.
- Do not auto-write migrated config during normal execution.

### Phase 2: Move templates out of normal config

- Implement built-in editor adapters.
- Keep custom command templates only as advanced config.
- Update examples and README to show only v2 minimal config.

### Phase 3: Prefer loopback + forwarding

- Change the recommended setup to a loopback daemon endpoint.
- Treat direct LAN/Tailscale mode as advanced.
- Default daemon bind should be `127.0.0.1:3339`, not `0.0.0.0:3339`.

### Phase 4: Explicit migration command

- Add `rcode config migrate` if automatic conversion is needed.
- Normal `rcode` and `rcode-server` startup should validate and report clear
  errors, not rewrite config.

### Phase 5: Deprecate v1 fields

- Warn when using `hosts.server.primary`, `hosts.server.fallback`, legacy
  `network.primary_host`, `fallback_editors`, and unified client/server config.
- Remove them only after a documented compatibility window.

## Compatibility mapping

During the transition, v1 fields can be mapped in memory:

```text
hosts.server.primary -> endpoint: http://<host>:3339
default_editor       -> editor
network.timeout      -> timeout
```

Avoid preserving the full v1 behavior in v2. In particular, do not keep the
primary/fallback host model as part of the minimal path.

## Security defaults

- Default daemon bind: `127.0.0.1:3339`.
- Use bearer token when exposing the daemon beyond loopback.
- Direct LAN/Tailscale mode must be explicit.
- Avoid unauthenticated `0.0.0.0` as a default.

## Desired outcome

The normal user-facing setup should be explainable as:

```text
Start rcode-server locally.
Forward a port from the remote machine to it.
Run `rcode .` remotely.
```

The normal config should be explainable as:

```yaml
endpoint: http://127.0.0.1:17339
editor: cursor
```

Everything else should be advanced configuration.
