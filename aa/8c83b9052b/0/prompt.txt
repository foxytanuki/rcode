# Light Mode — Plan → Implement → Selective Review

Lightweight pipeline that skips PRD creation. Parses plan mode output (or direct instructions), breaks into tasks, dispatches parallel subagents, and only reviews what needs reviewing.

**Core principle: minimum overhead, maximum parallelism.**

---

## Step 0: Input Detection

Determine the source of work:

1. **Plan mode output** (default): Check conversation context for a recently approved plan. Parse it.
2. **Direct argument** (`/light add auth middleware and rate limiter`): Use the argument as task description. Do your own quick task breakdown.
3. **Resume** (`/light resume`): Read existing `tasks/light/state.json` and continue from where we left off.
4. **No input, no plan, no state**: Error — tell user to either use plan mode first or provide a description.

---

## Step 1: Task Breakdown

Parse the plan/instruction into a flat task list. This is NOT a PRD — keep it minimal.

### From Plan Mode Output

Plans typically have numbered steps, phases, or bullet points. Convert each actionable step into a task:

```
Plan step: "1. Add Zod schemas for user input validation in src/schemas/"
→ Task: { id: "T1", title: "Add Zod schemas for user input", files: ["src/schemas/"], category: "schema" }
```

### From Direct Instructions

If no plan exists, analyze the instruction yourself:
1. Identify distinct implementation units
2. Infer dependencies from natural layer order
3. Each task must be completable in one focused pass

### Dependency Rules

Infer `dependsOn` automatically:
- Schema/type tasks → no deps
- Backend logic → depends on schema tasks it uses
- Frontend → depends on backend it consumes
- Tests → depends on the code they test
- Config/infra → usually no deps

### Review Necessity (Auto-Decide)

For each task, set `needsReview` based on category:

| `needsReview: true` | `needsReview: false` |
|---|---|
| DB schema / migration changes | UI component additions |
| Auth / security / permissions | Styling / layout changes |
| Structural changes (3+ files, new patterns) | Config / env changes |
| API contract changes (breaking) | Test additions |
| Final task (always — full integration check) | Extending existing patterns |

If unsure, default to `false`. Over-reviewing defeats the purpose.

### State File

Write to `tasks/light/state.json`:

```json
{
  "source": "plan-mode|direct|resume",
  "branch": "light/<feature-kebab>",
  "runner": "codex-implementer",
  "tasks": [
    {
      "id": "T1",
      "title": "Add Zod schemas for user input",
      "description": "Create validation schemas in src/schemas/ for...",
      "dependsOn": [],
      "needsReview": false,
      "status": "pending",
      "notes": ""
    }
  ]
}
```

**Minimal fields only.** No priority (order = priority), no acceptance criteria, no model overrides.

### Runner Detection

2. Create worktree at `.worktree/light/<feature-name-kebab>/` from base branch

If the plan involves only 1-2 simple tasks, skip worktree — work directly on a new branch.

## Step 3: Execute

### Task Selection

A task is **ready** when:
- `status == "pending"`
- All tasks in `dependsOn` have `status == "done"`

### Dispatch Strategy

```
ready_tasks = find all ready tasks

if len(ready_tasks) == 0 and incomplete tasks remain:
  → deadlock, stop and report

if len(ready_tasks) == 1:
  → dispatch single worker on main branch (no sub-worktree)

if len(ready_tasks) >= 2:
  → dispatch workers in parallel with worktree isolation
```

### Parallel Worktree Isolation

Same pattern as neoralph:
```bash
# For each parallel task
worker_branch="light/<feature>/workers/<task-id>"
worker_dir=".worktree/light/<feature>/workers/<task-id>"
mkdir -p "$(dirname "$worker_dir")"
git worktree add -b "$worker_branch" "$worker_dir" "light/<feature>"
```

After parallel workers complete:
1. Merge sub-branches back in task order
2. Cleanup worktrees and branches

### Worker Prompt

Spawn via Task tool:

```
subagent_type: <resolved runner>
model: opus (or inherit)
prompt:
  You are implementing one task in a lightweight pipeline.

  Branch: <branch>
  Working directory: <absolute path>

  Task: <id> - <title>
  Description: <description>

  Instructions:
  1. Read existing code to understand patterns
  2. Implement the task (focused, minimal changes)
  3. Run quality checks (typecheck/lint/test as applicable)
  4. If checks pass, commit: "feat(<scope>): <title>"
  5. If checks fail, fix before committing
```

For `codex-implementer` workers, the prompt additionally instructs to use `mcp__codex__codex` with `sandbox: "workspace-write"`.

### Post-Worker Verification

After each worker returns:
1. `git status --short` — check for uncommitted work
2. If uncommitted files from the worker: stage and commit
3. If worker already committed: verify with `git log -1 --oneline`
4. Update `state.json`: set `status: "done"`, add notes

### Error Handling

- Worker fails → retry once with error context, then mark `status: "failed"` and continue
- Merge conflict → attempt auto-resolve, if not possible mark failed
- Max 2 retries per task (lighter than neoralph's 3)

---

## Step 4: Selective Review

Only runs if ANY completed task has `needsReview: true`.

**If no tasks need review → skip entirely and go to Step 5.**

### Review Scope

Don't review everything — only the diff from tasks with `needsReview: true`:
```bash
# Get commits for review-worthy tasks only
git log --oneline main..<current-branch> -- <files from needsReview tasks>
```

### Review Execution

Spawn `codex-reviewer` subagent (same as neoralph):
```
subagent_type: codex-reviewer
model: sonnet
prompt:
  Review the following changes. Focus on: correctness, security, breaking changes.
  Do NOT review: style, naming, minor improvements.

  Scope: [list of review-worthy tasks and their files]
  Branch diff: git diff main..HEAD
```

### After Review

- Critical/High issues → spawn a fix worker, then continue
- Medium/Low → note in summary, don't block
- **No second review round.** Fix once and move on.

---

## Step 5: Completion

1. Verify final state: all tasks `done` or `failed`
2. Run final quality check (typecheck + test if applicable)
3. Print summary:
   ```
   Light Mode Complete
   ──────────────────
   Tasks: X/Y completed (Z failed)
   Review: [skipped | passed | N issues fixed]
   Branch: light/<feature>
   Commits: <count>
   ```
4. Clean up `tasks/light/state.json` (delete it — git log is the real record)
5. Do NOT push or create PR unless asked

---

## Autonomy Rules

Same as neoralph — do NOT ask the user unless:
- Missing credentials or API keys
- Ambiguous requirements
- Destructive operations

For everything else, decide and act. This is light mode — speed is the point.

| Aspect | NeoRalph | Light Mode |
|---|---|---|
| Input | PRD (markdown) → prd.json | Plan mode output or direct text |
| State | prd.json (rich schema) | state.json (minimal) |
| Review | Every phase boundary | Only risky tasks + final |
| Progress | progress.txt (detailed) | git log (commits are the record) |
| Worktree | Always for feature branch | Skip for simple tasks |
| Recovery | Resume from prd.json | Resume from state.json |
| Overhead | High (PRD + convert + preflight) | Low (parse + go) |
